That QR code above wasn't made by a library. Every dark and light square was placed by code I wrote from scratch — finder patterns, error-correction bytes, data zig-zag, mask selection — and it scans to this site's URL. Point your phone at it.

The QR code is the single richest tool in my [Coding Challenges Deep Dive](https://github.com/singhhimanshu004/coding-challenges-deepdive), because "make a QR code" secretly means "implement finite-field algebra and Reed–Solomon error correction." This is the post I wish I'd had before starting: the real pipeline, and the two ideas that make it work.

## The pipeline

```
text → mode analysis → smallest fitting version → bit encoding
     → terminator + padding → split into blocks → Reed–Solomon ECC
     → interleave → paint into matrix → pick best mask → done
```

The front half turns your text into a precise stream of bytes (*codewords*). The back half paints those bytes into the 2D grid in the exact order and orientation the standard demands. The magic — the part that lets a scanner read a QR code that's scratched, smudged, or partially covered — lives in the middle: **Reed–Solomon error correction**. And Reed–Solomon runs on math that doesn't behave like ordinary arithmetic.

## GF(256): arithmetic that never overflows a byte

QR's error correction does all its math in a *finite field* called GF(256): the 256 byte values `{0..255}` form a closed system where you can add, subtract, multiply, and divide without ever leaving the set.

Why not ordinary integer math? Error correction needs two things integers can't give you in a byte: every non-zero element must have a multiplicative inverse (so division always works), and arithmetic must never overflow. `200 * 200` overflows a byte; `6` has no integer inverse. A finite field hands you both for free.

The construction has two beautiful quirks:

- **Addition *is* XOR.** There are no carries. In GF(256), `3 + 1 == 2`.
- **Multiplication is a slide rule.** Every non-zero element is a power of the generator `g = 2`, so you precompute two tables — `EXP[i] = 2^i` and its inverse `LOG` — and then `a*b = EXP[LOG[a] + LOG[b]]`. Multiplication becomes *addition of exponents*.

```python
def _build_tables():
    x = 1
    for i in range(255):
        EXP[i] = x
        LOG[x] = i
        x <<= 1                 # multiply by the generator (2)
        if x & 0x100:           # overflowed 8 bits?
            x ^= PRIMITIVE      # carry-less reduction mod x^8+x^4+x^3+x^2+1
    for i in range(255, 512):   # mirror so EXP[i+j] never indexes out of range
        EXP[i] = EXP[i - 255]

def mul(a, b):
    if a == 0 or b == 0:
        return 0
    return EXP[LOG[a] + LOG[b]]
```

The `PRIMITIVE = 0x11D` constant is QR's irreducible polynomial, `x^8 + x^4 + x^3 + x^2 + 1`. The doubled `EXP` table is a small trick that lets `LOG[a] + LOG[b]` (which can reach 508) index in without a modulo in the hot path. Once this field exists, everything else is "just" polynomials over it.

## Reed–Solomon is polynomial long division

Here's the idea that took me longest to *feel*. Treat your block of data bytes as the coefficients of a polynomial `M(x)`. To add `n` error-correction codewords, compute the remainder of `M(x) · x^n` divided by a fixed *generator polynomial* `g(x)` — all in GF(256). Those `n` remainder bytes *are* the EC codewords.

Why does that correct errors? The generator `g(x)` is built to have `n` roots at consecutive powers of the field generator. Because the transmitted codeword is a multiple of `g(x)`, it evaluates to zero at every one of those roots. A decoder checks those evaluations — the "syndromes" — and any non-zero result reveals *and locates* the damage. (I only implement the encoder; the matching decoder is a whole post of its own.)

Building the generator is just folding `(x - 2^i)` factors together with polynomial multiply:

```python
def generator_poly(n):
    g = [1]
    for i in range(n):
        g = poly_mul(g, [1, galois.EXP[i]])   # (x - 2^i); subtraction is XOR
    return g
```

And the encode step is synthetic long division, done in place with the standard LFSR-style loop — `O(len(data) · n)`:

```python
def encode(data, n_ec):
    gen = generator_poly(n_ec)
    remainder = list(data) + [0] * n_ec
    for i in range(len(data)):
        coef = remainder[i]
        if coef == 0:
            continue
        for j in range(1, len(gen)):           # skip gen[0], which is 1
            remainder[i + j] ^= galois.mul(gen[j], coef)
    return remainder[len(data):]               # the remainder = EC codewords
```

## Spreading the damage: interleaving

A subtle, clever bit of the standard: QR doesn't store one block's data then its EC bytes contiguously. It **interleaves** codewords across blocks, column-major. The reason is physical — a coffee-cup ring or a thumb covers a *contiguous region* of the symbol. Interleaving means that localized smudge spreads its damage thinly across *every* block, and each block can self-heal a little, rather than one block being destroyed entirely and exceeding its correction budget.

## Painting the grid (and the parts that aren't data)

A QR symbol isn't all payload. Fixed *function patterns* orient the scanner: the three big **finder** squares in the corners, the **timing** lines that bridge them, alignment patterns, and a lone dark module. The data-placement walk has to skip all of them, plus the reserved strips for format and version info. Then it threads the remaining cells in the standard upward/downward **zig-zag**, dropping codeword bits as it goes.

The format and version strips aren't left bare either — they carry their own **BCH codes** (a cousin of Reed–Solomon) so a scanner can still recover which mask and error-correction level were used even if those few modules are damaged.

## Eight masks, four penalties

A naive bit layout can accidentally produce big solid blocks, or patterns that look like the finder marks, and confuse a scanner. So QR XORs the *data* region (never the function patterns) with one of **eight** fixed mask patterns — simple functions of each module's `(row, col)`:

```python
MASK_FUNCS = [
    lambda r, c: (r + c) % 2 == 0,
    lambda r, c: r % 2 == 0,
    lambda r, c: c % 3 == 0,
    lambda r, c: (r + c) % 3 == 0,
    # ...four more...
]
```

You apply *all eight*, score each with four penalty rules (long same-color runs, 2×2 blocks, finder-like sequences, and overall dark/light imbalance), and keep the lowest-scoring — most readable — result. The mask isn't a detail; it's the difference between a code that scans instantly and one your phone struggles with.

## Why a QR code is the perfect "hard" project

Most from-scratch projects teach one idea. A QR code generator makes you build a small algebra library, an error-correction coder, a 2D layout engine, and a scoring heuristic — and then *the real world grades your work instantly*: either a phone camera reads it, or it doesn't. There's no arguing with a scanner. When my generated code finally decoded on the first try, after the Galois tables and the polynomial division and the mask scoring all had to be simultaneously correct, it was the most satisfying green checkmark of the whole project.
